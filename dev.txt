
  ● Reflector containing both MsgEnv and transport
      // Transport = anything with .start() and .write(std::string); read bytes via the cb you give it.
    // MsgEnv    = anything with .publish(const char*) and .subscribe(cb, void*).
    template <typename Transport, typename MsgEnv>
    class Reflector {
        duplicate          _dedup;
        std::vector<char>  _buf;
        Transport&         _transport;
        MsgEnv&            _msgenv;
    public:
        Reflector(Transport& t, MsgEnv& m) : _transport(t), _msgenv(m) {}

        // asio -> msgenv  (the transport's read callback points here)
        void on_read(const char* data, std::size_t len) {
            l4end::deframe(&_buf, data, len, [this](std::string msg){
                if (_dedup.exists(msg)) return;
                _msgenv.publish(msg.c_str());
            });
        }
        // msgenv -> asio
        bool on_msg(std::string msg) {
            _dedup.cache(msg, t_entry_context(30s));
            _transport.write(l4end::enframe(msg));
            return true;
        }
        void start() {
            _msgenv.subscribe(&trampoline, this);   // void* user-data carries `this`
            _transport.start();
        }
    private:
        static void trampoline(const char* m, void* self) { static_cast<Reflector*>(self)->on_msg(m); }
    };

● Worth mentioning: the names are actually still perfectly regular — the grammar just went from one token to two. Every new unit is
  asio_<kind>_<role>:

  asio_tcp_client       (kind=tcp,       role=client)
  asio_udp_client       (kind=udp,       role=client)
  asio_broadcast_server (kind=broadcast, role=server)
  asio_rr_server        (kind=rr,        role=server)

  So you can get the clean interpolation back — it'd just be a two-token key instead of one:

  cpp-${TRANSPORT}.git/asio_${KIND}_${ROLE}.cpp

  with a KIND knob (tcp/udp/broadcast/rr). That reads as nicely as the old asio_${ROLE} and is strictly more expressive.

  But I'd argue the sadness is pointing at the right thing and the cure would be worse here. The old asio_${ROLE} was clean precisely because the
  names were under-specified — one token sufficed only because client/server was the whole story. That under-specification is exactly what these
  renames set out to kill (your call, and the right one). So the loss of the one-token map isn't a regression in the naming; it's the bill for the
  specificity.

  And the KIND knob recovers the elegance only on paper: this reflector ships exactly two variants (tcp_client, broadcast_server). A second knob
  whose four combinations are never all exercised is machinery pretending the build is more general than it is — and the asymmetry bites immediately,
  since KIND can't default sensibly without knowing ROLE (client wants tcp, server wants broadcast), which lands you right back at an if/else, now
  spread across two knobs.

  So my honest read: the explicit ROLE→unit if/else is the more truthful encoding now — it says "there are two supported builds, here they are"
  instead of implying a 2×N grid. The interpolation was prettier; the if/else is more honest about what actually exists.
